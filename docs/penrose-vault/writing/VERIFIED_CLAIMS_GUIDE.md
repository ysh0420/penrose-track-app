# Verified Claims Guide

## Purpose

Use this guide to decide what can be written in a Penrose article, report draft, or client-facing note.

## Claim Levels

| Level | Meaning | Allowed Use |
| --- | --- | --- |
| Primary-source fact | Directly supported by filings, disclosures, company materials, or official sources | Can be used with source date and context. |
| Platform metric | Derived from an approved data source or Supabase output | Can be used if provenance and calculation context are clear. |
| Brain diagnostic | Generated by Penrose Brain logic | Can be used as internal diagnostic unless approved for external wording. |
| Interpretation | Analyst view based on verified claims | Must be clearly framed as interpretation. |
| Missing data | Required input not available or incomplete | Must remain missing. Do not fill. |

## Required Metadata

- Source name.
- Source date.
- Retrieval or run date.
- Coverage limitation.
- Whether the claim is public, internal, or client-specific.

## Rules

- Use current Supabase outputs for platform state.
- Use GitHub for code and docs state.
- Use source packs for writing inputs.
- Keep missing values visible.
- Separate facts from interpretation.
- Avoid hard claims when coverage is partial.

## Not Enough Evidence

Do not use a claim when:

- It is only remembered from chat.
- It depends on stale page copy.
- It is not supported by a source pack.
- It requires sector RS, peer basket RS, or global lead-lag values that are missing.
- It exposes internal recommendation, portfolio action, or raw paid-data-derived content.

## Suggested Language For Gaps

- "Coverage is incomplete for this field."
- "The current source pack does not support that claim."
- "This remains a diagnostic signal, not a portfolio action."
- "The available evidence is partial and should be refreshed before publication."
